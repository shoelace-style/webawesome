# Webawesome skill — hand-authored references

This directory is the **hand-maintained source** for static reference files that ship inside the
auto-generated `webawesome` agent skill. The skill itself is generated by `scripts/agent-skill.js`
from the Custom Elements Manifest and rendered docs HTML; the files here are the small set of
references that don't fit that pipeline because they're not derivable from the library.

> **Note:** This `README.md` is for maintainers. It is **not** shipped with the skill; the generator
> copies hand-authored references by name (see the `fs.copyFileSync` call in `agent-skill.js`) and
> this README isn't in that list.

## What lives here

```
scripts/agent-skill/                  ← EDIT THESE (source of truth, in git)
  choosing-components.md                decision tree by user intent — helps AI tools pick the right
                                        component instead of guessing from names
        │
        ▼  copied verbatim by scripts/agent-skill.js into the skill's references/
dist/skills/webawesome/references/    ← GENERATED, alongside themes.md, support.md, components/, etc.
dist-cdn/skills/webawesome/references/
```

The companion `webawesome-design` skill works the same way — its source is at
`scripts/design-skill/` with its own README; the maintainer patterns mirror each other.

## How it ships

`scripts/agent-skill.js` runs as part of the eleventy build. After it writes `SKILL.md` and copies
component docs, it copies every `.md` file in this directory into the output `references/` dir
verbatim, alongside the generated `themes.md`, `support.md`, `installation.md`, `usage.md`,
`form-controls.md`, `customizing.md`, `localization.md`, the per-component `components/*.md`, and the
framework guides under `frameworks/`.

The generated `SKILL.md` also gets two pointers to `choosing-components.md`: a callout under
`## Components` ("Not sure which one to pick?") and the lead entry in
`## Reference Documentation` ("start here if you're unsure which component fits"). When adding a new
static reference, update both surfaces so it's discoverable.

## To make a change

1. Edit the relevant Markdown file in this directory.
2. Re-copy into dist via a full build (`npm run build`).
3. Run Prettier so source and dist stay byte-identical and formatted:

   ```bash
   npx prettier --write 'scripts/agent-skill/**/*.md'
   ```

4. Run the skill verifier:

   ```bash
   npm run verify:skills
   ```

**Never edit `dist/skills/webawesome/references/` directly**; every build overwrites it from this
source (and from the CEM-driven generators for everything else).

## The verifier and what it checks here

`scripts/verify-skills.js` cross-checks files in this directory against the library so silent drift
can't ship. Specifically for `choosing-components.md`:

- Every `<wa-*>` tag mentioned in a table must exist in either `packages/webawesome/src/components/`
  (free) or `packages/webawesome-pro/src/components/` (Pro)
- Rows marked `**(Pro)**` must point at the Pro package; unmarked rows must point at the free package
- Every `<wa-tag attr>` and `<wa-tag attr="value">` cited must reference a real attribute per the CEM
  (caught the case where `<wa-tag removable>` should have been `<wa-tag with-remove>`)
- Relative markdown links must resolve, with a small allowlist for generator-output siblings (e.g.
  `form-controls.md`, `themes.md`) that don't exist at source but will exist at runtime in the same
  `references/` directory

If the verifier fails after a library change, fix the citation (or the library) before merging.

## Keeping content accurate

The verifier catches structural drift (tag/attribute names, Pro/Free classification, links). Some
content still needs hand-maintenance because it's curated rather than derivable:

| If you change…                                                                               | Update                                                                |
| -------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| Component selection heuristics (e.g. dropdown vs. select, switch vs. checkbox)               | The relevant intent section in `choosing-components.md`               |
| The set of Pro-only components                                                               | The intent tables that cite them + the "A note on Pro" footer         |
| A component is renamed, added, or removed                                                    | Tables in `choosing-components.md` (verifier will fail until updated) |
| A component's attribute that's named in the tree (e.g. `multiple`, `with-remove`, `variant`) | The attribute usage in the table (verifier will catch typos)          |

## Conventions

- **Organize by user intent, not by component name.** The skill exists to fight the most common agent
  mistake — picking the wrong component for the job. Every section should read as "you want X → use
  this." If a section is named after a component, it's in the wrong skill (it belongs in the
  generated component reference, not here).
- **Tables hold the authoritative classification.** The verifier reads table rows for tag and Pro
  marking. Prose mentions are looser. If you want a check to apply, put it in a table row.
- **Mark Pro with the literal `**(Pro)**` token in the row.** That's what the verifier looks for.
- **Surface confusions explicitly.** Each section should call out the most common recurring confusion
  (dropdown vs. select, callout vs. toast, tag vs. badge, dialog vs. drawer, etc.). Those callouts
  are the highest-value content here.
- **Point at the companion `webawesome-design` skill** for layout, theming, and visual composition
  guidance. Don't duplicate it here.
